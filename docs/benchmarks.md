# Benchmarks

`rustysoup` benchmarks are designed to be reproducible, realistic, and easy to audit.

The benchmark harness is [benches/benchmark_parsers.py](../benches/benchmark_parsers.py). It compares `rustysoup` with widely used Python HTML parsers:

- `rustysoup`
- BeautifulSoup with `html.parser`
- BeautifulSoup with `lxml`
- direct `lxml.html`
- `html5_parser`
- `selectolax` with Lexbor
- `selectolax` with Modest

Comparison parsers are optional. Missing or unusable packages are reported as skipped in the generated report.

## Recommended Setup

Build `rustysoup` in release mode before collecting benchmark numbers:

```bash
python -m pip install beautifulsoup4 selectolax html5-parser
python -m pip install --force-reinstall --no-binary lxml lxml
maturin develop --release
```

The source-built `lxml` step keeps `lxml` and `html5_parser` on the same local `libxml2`, allowing both parser backends to run in the same comparison.

## Quick Smoke Run

```bash
python benches/benchmark_parsers.py --suite smoke --documents 5 --items-per-doc 20 --iterations 3 --warmups 1
```

List available parsers and workloads:

```bash
python benches/benchmark_parsers.py --list
```

## Standard Run

Use a real corpus whenever possible:

```bash
python benches/benchmark_parsers.py \
  --input-dir ./html-corpus \
  --suite standard \
  --max-files 750 \
  --iterations 25 \
  --warmups 5 \
  --min-time 0.25 \
  --report benchmark-report.md \
  --json-output benchmark-report.json
```

For memory:

```bash
python benches/benchmark_parsers.py \
  --input-dir ./html-corpus \
  --max-files 750 \
  --iterations 10 \
  --warmups 3 \
  --memory \
  --memory-iterations 3
```

Memory numbers are process-local peak RSS deltas measured in isolated subprocesses. They are useful for parser-level comparisons, not low-level heap accounting.

## Workload Groups

Suites:

- `smoke`: fast parser/query sanity check for local development.
- `standard`: headline suite for regular comparisons.
- `full`: every available operation, including lower-level diagnostics.

Parse workloads:

- `parse`: parse and discard each tree.
- `parse_hold`: parse and retain all trees until the operation completes.
- `parse_threaded`: parse through a `ThreadPoolExecutor` with `--threads`.

Cold parse+query workloads:

- `parse_select_links`
- `parse_select_product_links`
- `parse_find_divs`
- `parse_extract_text`
- `page_summary`
- `link_graph`: collect and classify hrefs against each document URL.
- `article_extract`: title, headings, paragraphs, and article/main links on article-like pages.
- `product_extract`: product-ish cards, links, prices, images, and text on ecommerce pages.
- `table_extract`: table, row, cell, and header structure on table-heavy pages.
- `form_extract`: forms, labels, controls, names, types, and boolean attrs on form-heavy pages.
- `parse_materialize_links`: materialize link nodes and access Python-facing properties.
- `parse_selector_stress`: run common selectors used in crawler/scraper workloads.

Hot query-only workloads:

- `reuse_select_links`
- `reuse_select_product_links`
- `reuse_find_divs`
- `reuse_extract_text`
- `reuse_materialize_links`
- `reuse_selector_stress`

Hot workloads preparse the corpus once outside the timed query loop. This separates selector/traversal speed from parse speed.

Category-aware workloads use the corpus manifest generated by `benches/collect_corpus.py`. For example, `product_extract` runs against ecommerce pages, so documents/sec and MiB/sec are calculated against the relevant working set.

## Reading Results

The Markdown report sorts by median time, not best time. Each row includes:

- median, best, mean, and p95 latency
- documents/sec and MiB/sec
- relative speed versus the baseline parser
- relative standard deviation
- deterministic checksum
- optional peak RSS delta
- the number of documents used by that operation

Comparable workloads should produce matching checksums. If checksums differ, the report highlights the parser groups so behavior can be inspected alongside performance.

Text extraction results can differ across parsers because each library makes different choices around whitespace, scripts, styles, and malformed markup.
